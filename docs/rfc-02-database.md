## Decisions

### Supabase CLI — dev dependency, not global

The CLI docs and existing product request specified `npm install -g supabase`. Global
install is not supported in this environment. The correct approach (per the official
[supabase/cli README](https://github.com/supabase/cli#install-the-cli)) is to install
as a dev dependency and invoke via `npx`:

```bash
npm install --save-dev supabase
npx supabase login
```

The spec files (`2-database-setup.md`) have been updated to reflect this.

### Migrations applied manually via SQL Editor

The `supabase db execute` command does not accept a `--stdin` flag in current CLI
versions. The product request's CLI-based alternative was updated to use `--file`:

```bash
npx supabase db execute --file packages/db/migrations/001_extensions.sql
```

All six migrations were applied manually via the Supabase Dashboard SQL Editor.
The `2-database-setup.md` spec has been updated to correct the CLI commands.

### `api_usage_log` treated as required

The `api-first/02-database.md` spec marks `006_api_usage_log.sql` as optional.
It was implemented as required because `apps/web/lib/api/usage.ts` references the
table — omitting it would cause a runtime error the first time any API route calls
`logApiRequest()`.

### `usage.ts` uses `createServiceClient()`, not `createClient()`

The product request's code sample calls `createClient()` (publishable key). The spec
comment for `api_usage_log` states "inserted only via service role". In this project's
architecture, the secret key client (`createServiceClient()`) bypasses RLS, which is
the correct behaviour for a write-only internal table with no RLS write policy.
The implementation overrides the spec sample to use `createServiceClient()`.

## What Was Applied

### Migration 001 — Extensions
Enabled `pgvector`, `pg_trgm`, and `unaccent` extensions.

### Migration 002 — Core Tables
Created all 12 domain tables:
`members`, `documents`, `document_texts`, `document_authors`, `votes`, `vote_results`,
`speeches`, `budget_outcomes`, `manifestos`, `manifesto_statements`, `document_chunks`,
`ingestion_runs`.

### Migration 003 — Indexes
Full-text search indexes (Swedish dictionary), trigram index on member names, filtering
indexes on all high-cardinality columns, HNSW vector indexes on `manifesto_statements`
and `document_chunks` (m=16, ef_construction=64, cosine distance).

### Migration 004 — RLS
Row Level Security enabled on all 12 tables. Public `SELECT` policy on all data tables.
No public read on `ingestion_runs` (internal only).

### Migration 005 — Functions
`search_documents(query_text, query_embedding, doc_type, doc_rm, match_count)` —
hybrid search using a CTE that combines full-text rank (40%) and vector similarity (60%).

### Migration 006 — API Usage Log
`api_usage_log` write-only table with `endpoint`, `params_hash`, `duration_ms`,
`status_code` columns. Two indexes: by endpoint+date and by status+date.

## Verification

All three verification queries confirmed in the Supabase SQL Editor:

```sql
-- 13 tables in public schema
SELECT table_name FROM information_schema.tables
WHERE table_schema = 'public' ORDER BY table_name;

-- All data tables have RLS enabled
SELECT tablename, rowsecurity FROM pg_tables
WHERE schemaname = 'public' ORDER BY tablename;

-- search_documents function exists
SELECT proname FROM pg_proc WHERE proname = 'search_documents';

-- pgvector extension active
SELECT extname FROM pg_extension WHERE extname = 'vector';
```

TypeScript typecheck from monorepo root: zero errors.
