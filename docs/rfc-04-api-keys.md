## Summary

Implemented the full API key lifecycle: database schema, cryptographic key generation,
hashed storage, validation middleware, key-aware rate limiting, the public self-service
`POST /api/v1/keys/request` endpoint, and the OpenAPI spec scaffolding. Raw keys are
never persisted — only a SHA-256 hex hash and an 8-character display prefix. The key
is shown to the consumer once at creation time and is unrecoverable thereafter.
All gated endpoints in later product requests depend on `requireApiKey()` from this PR.
`tsc --noEmit` exits clean.

---

## Motivation

Product request 07 (`/api/v1/accountability`) is gated behind API key authentication.
The middleware and rate-limit infrastructure needed to exist first so that later route
handlers can simply call `requireApiKey(req)` without building the plumbing themselves.
The key request endpoint also lets developers self-provision credentials against the
live API without operator involvement.

---

## Decisions

### Migration slot 009 (not 007 as specced)

The product request calls for `007_api_keys.sql`. Migrations 007
(`007_voyage_dimensions.sql`) and 008 (`008_manifesto_statements_unique.sql`) were
already applied by PR-02/03. The `api_keys` table was therefore created in slot `009`
to preserve sequential ordering without gaps.

### `createServiceClient()` required for all `api_keys` operations

The spec's `auth.ts` sample calls `createClient()` (the anon/publishable-key client).
The `api_keys` table has RLS enabled with **no public read policy** — the design intent
is service-role-only access. Using `createClient()` would cause every key lookup to
return `null` silently. Both `auth.ts` and the `keys/request` route handler use
`createServiceClient()`.

### `request_count` increment corrected

The spec's fire-and-forget update sets `request_count: data.request_count` — a no-op
that never actually increments the counter. The implementation corrects this to
`request_count: data.request_count + 1`.

### Zod v4 `.issues` instead of `.errors`

The project uses Zod v4 (`"zod": "^4.0.0"` in `package.json`). In Zod v4 the
`.errors` array was renamed to `.issues`. The route handler uses `.issues[0]?.message`
accordingly.

### DB types updated manually

`packages/db/src/database.types.ts` is normally regenerated by `supabase gen types`.
Because the migration (`009_api_keys.sql`) has not yet been applied to the hosted
Supabase project, the CLI would not include `api_keys` in a fresh generation. The
`api_keys` Row/Insert/Update types were added manually so the codebase type-checks
now; they can be overwritten by a CLI regeneration once the migration is applied.

### `anon_account` limiter intentionally absent

The rate-limit table defines tiers `anon`, `free`, and `paid` × route classes
`general`, `search`, and `account`. Anonymous consumers are blocked at the auth layer
on account-class endpoints (they receive 401 before the rate limiter runs), so there
is no `anon_account` bucket in `limiters`. The `checkRateLimit` helper is only called
on account-class routes after `requireApiKey()` confirms a valid key.

## Architecture

```
POST /api/v1/keys/request
       │
       ├── RequestKeySchema (Zod) — email + description (min 20 chars)
       ├── createServiceClient() → api_keys.count WHERE email + is_active (max 3)
       ├── generateRawKey()      → "agora_" + 43 base64url chars (32 bytes entropy)
       ├── hashKey()             → SHA-256 hex (64 chars) — the only value persisted
       ├── extractPrefix()       → first 8 chars of body — for key identification
       └── INSERT api_keys { key_prefix, key_hash, email, description, tier='free' }
               └── returns { key, prefix, tier, message, usage, docs }
                   key shown ONCE — not stored, not recoverable

GET /api/v1/openapi
       └── generateOpenApiSpec() — OpenApiGeneratorV31 from registry definitions

lib/api/auth.ts  (used by gated routes in PR-07+)
       │
       ├── extract Bearer token from Authorization header
       ├── isWellFormed() — rejects malformed values without DB roundtrip
       ├── hashKey()      — hash for DB lookup
       ├── createServiceClient() → api_keys WHERE key_hash (single row)
       ├── is_active check → 403 if revoked
       ├── fire-and-forget UPDATE last_used_at + request_count+1
       └── returns ApiKeyContext { keyId, tier, rateLimitRpm }

lib/rate-limit.ts  (used by route handlers in PR-05+)
       │
       ├── 8 Upstash Ratelimit instances (slidingWindow, 1 minute)
       │   anon:  general=30, search=10
       │   free:  general=120, search=30, account=10
       │   paid:  general=600, search=120, account=60
       └── checkRateLimit(req, routeClass, keyId?, tier?)
               └── identifier = keyId (keyed) or client IP (anonymous)
```
